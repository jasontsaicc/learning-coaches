▸01 One-Liner · Trade-off · Scale

底層直覺 先講，再核對

Cache 把常用的資料放在更快、更近的儲存層（如 Redis），擋住大部分 request 不用打到 DB，大幅降低 latency 和 DB 負載。

Trade-offCache 的三個關鍵 trade-off 你能講出來嗎？
  • ▸Cache-Aside write 時「刪除」而不是「更新」cache：避免 race condition 導致 cache 和 DB 不一致
  • ▸Write-Through vs Write-Behind：一致性 vs 寫入速度。資料不能丟用 Write-Through，高寫入量可容忍短暫遺失用 Write-Behind
  • ▸先寫 DB 再刪 cache vs 先刪 cache 再寫 DB：先寫 DB 再刪 cache 比較安全，反過來有 race condition 風險
Scale什麼時候該加 cache？目標是什麼？

當 DB query latency 成為瓶頸（P99 > 50ms）或 QPS 超過單機 DB 上限 → 加 Redis cache layer，目標 hit ratio > 90%。

DevOpsElastiCache 要監控什麼？hit ratio 突降代表什麼？
  • ▸ElastiCache (Redis) 監控重點：cache hit ratio、eviction rate、memory usage
  • ▸Cache hit ratio 突然下降 > 10% → 可能是 key pattern 改變或 deploy 導致 cold start
  • ▸CloudFront 的 cache hit ratio 可在 CloudWatch 看到
▸02 核心概念 · Why It Works

Cache 為什麼有效 locality

回想Cache 為什麼有效？靠哪個性質？什麼情況下它沒用？

靠 locality of access（存取局部性）：

  • ▸Temporal locality：最近存取的資料很可能很快再被存取
  • ▸80/20 rule：80% 的 request 打 20% 的資料（hot data）
  • ▸如果每個 request 都存取不同資料，cache 就沒用了
Effective Latency 公式（原始筆記）

Effective Latency = (Hit Rate × Cache Latency) + (Miss Rate × DB Latency)
90% hit, cache 1ms, DB 20ms = (0.9 × 1) + (0.1 × 20) = 2.9ms ← vs 20ms without cache

深掘公式算出 90% hit → 2.9ms。如果 hit 從 90% 掉到 80%，latency 變多少？為什麼 miss rate 一點點變化影響這麼大？蘇格拉底深掘 · 非原始筆記

80% hit = (0.8×1) + (0.2×20) = 0.8 + 4 = 4.8ms，從 2.9 跳到 4.8，差約 65%。

原因：miss 的代價（20ms）是 hit（1ms）的 20 倍，所以 effective latency 幾乎被「miss rate × DB latency」這一項主宰。hit 從 90→80，miss 從 10%→20% 是翻倍，那條昂貴項就翻倍。這跟 Day 27 那張「hit ratio 跌 5% → DB 流量翻倍」是同一件事的兩面：miss rate 同時決定 latency 和 DB 成本。所以 hit ratio 不是虛榮指標，是 latency 和成本的共同槓桿。

▸04 讀取策略 · Read Patterns

Cache-Aside vs Read-Through 最常考

Cache-Aside讀流程、寫流程？為什麼刪不更新？順序為什麼重要？

App 自己管 cache，DB 和 cache 之間沒有直接連結。

Read flow:
App → Cache (hit?) → Yes → 回傳
                   → No (miss) → App → DB → App → 寫回 Cache → 回傳

Write flow:
App → 寫 DB → 刪除 Cache key
  • ▸寫入時刪除而不是更新：避免兩個 thread 同時更新造成 race condition
  • ▸順序很重要：先寫 DB 再刪 cache（反過來有 race condition 風險）
  • ▸優點：只 cache 真正被讀的資料（lazy），不浪費記憶體
  • ▸缺點：第一次 read 一定 miss（cold start）
Read-Through跟 Cache-Aside 差在哪？一個比喻？

Cache 自己管 DB 讀取，App 只跟 Cache 說話。

App → Cache (hit?) → Yes → 回傳
                   → No (miss) → Cache → DB → Cache (store) → 回傳
  • ▸比喻：Cache-Aside 是自己去超市買，Read-Through 是叫管家去買
  • ▸App code 更簡潔，但需要 cache library 支援（如 Guava Cache, AWS DAX）
深掘為什麼寫入要「刪」cache 不「更新」？如果改成更新，兩個 thread 並發寫同一 key 會發生什麼？蘇格拉底深掘 · 非原始筆記

兩個寫入：A 要把值改成 X、B 改成 Y。DB 那邊有鎖會定序（假設最後落地是 Y）。但 cache 的 update 沒鎖：A 算完去寫 cache(X)、B 算完去寫 cache(Y)，這兩個 update 到達 cache 的順序可能跟 DB 的提交順序相反 → DB 是 Y、cache 卻被 A 慢一步蓋成 X → 髒。

改成「刪」就沒這問題：兩個寫都只是刪 key，下一次讀 miss 會從 DB 拿到正確的 Y 回填。刪是冪等的、不帶值，所以不可能把舊值寫進去。往回扣「先寫 DB 再刪」那個順序題。

▸05 寫入策略 · Write Patterns

Write-Through vs Write-Behind

Write-Through流程、優缺點、適合場景？
App → Cache (write) → Cache → DB (write) → 回傳 success
  • ▸優點：Cache 和 DB 永遠一致
  • ▸缺點：write latency 高（要等兩邊都寫完）、可能 cache 了沒人讀的資料
  • ▸適合：資料一致性要求高、讀多寫少
Write-Behind流程、優缺點、適合場景？最大風險是什麼？
App → Cache (write) → 回傳 success（馬上！）
      Cache → (async batch) → DB（之後再寫）
  • ▸優點：write latency 極低、batch write 減少 DB 壓力
  • ▸缺點：Cache 掛了 → 未 flush 的資料遺失
  • ▸適合：高寫入量、可容忍短暫遺失（如 game score、analytics、page view counter）
比較三種寫入策略：流程 / latency / 一致性 / 風險？
Pattern寫入流程Latency一致性風險
Cache-AsideApp → DB → 刪 cache中最終一致Cold read after write
Write-ThroughApp → Cache → DB（同步）高強一致Write slow
Write-BehindApp → Cache → DB（異步）低弱一致Data loss on crash

搭配：Read-Through + Write-Through → App 完全不碰 DB，只跟 Cache 互動（例：AWS DAX）。

深掘Write-Behind 說 cache 掛了沒 flush 的資料會掉。可是 DB 自己的 WAL 也是「先寫記憶體後落盤」，為什麼一個我們說危險、一個我們信任？蘇格拉底深掘 · 非原始筆記

差在「落盤承諾」。DB 的 WAL 是：先把變更順序寫進持久化的 log（fsync 落盤）才回 client success，記憶體裡的 page 之後慢慢刷；crash 後用 log 重放，已確認的寫不丟。

Write-Behind 的 cache 通常是：寫進揮發性記憶體就回 success，還沒落任何持久層，crash = 真的不見。所以同樣「先記憶體後落盤」，WAL 在回 success 前已經有持久紀錄，write-behind 沒有。要讓 write-behind 安全，就得給 cache 一個持久 log（等於把它變成半個 DB）。這就是為什麼 write-behind 只敢用在「可容忍少量遺失」的場景。

▸06 Eviction Policies

滿了踢誰 LRU / LFU / TTL

回想LRU / LFU / TTL 各踢誰、各適合什麼？LRU 和 LFU 的關鍵差別？
Policy踢誰適合
LRU (Least Recently Used)最久沒被用的最常用，通用場景（Redis 預設）
LFU (Least Frequently Used)被用最少次的有明確 hot data 不想被偶爾 cold request 洗掉（如熱門商品頁）
TTL (Time To Live)過期的資料有明確過期時間
  • ▸LRU 看「最後一次」存取時間，LFU 看「總共」存取次數
  • ▸LFU 問題：過去很熱門但現在沒人用的 key 會一直留著 → 解法：加 decay 衰減 count
  • ▸Redis 預設：maxmemory-policy allkeys-lru
深掘一個剛被瘋傳的新 key（才被讀幾次但正在狂熱），記憶體滿要 evict 時，LRU 和 LFU 誰比較可能誤殺它？蘇格拉底深掘 · 非原始筆記

LFU 比較可能誤殺。LFU 看「累積次數」，新 key 再熱次數也還小，容易被一個「歷史總次數高但其實已退燒」的老 key 比下去而被踢。LRU 看「最後一次存取」，剛被讀過的新熱 key 排在最不該踢的那端，反而保得住。

這正是 LFU 的盲點（要配 decay 衰減老 key 的計數才公平），也是為什麼 Redis 預設給 LRU。連到 Day 27 的「熱 key」：一條病毒短碼就是這種「新但極熱」的 key，cache 策略要保得住它。

▸07 Invalidation · The Hard Problem

失效與驚群 cache 最難的一塊

回想三種 invalidation 策略？最常見的組合是哪兩個搭？
Strategy做法優缺點
TTL-based設過期時間，自動刪最簡單，但 TTL 內可能 stale
Event-based資料變更時主動刪 cache即時一致，但實作複雜
Version-basedkey 帶版本號 user:123:v5不用刪舊的，但 key 膨脹

最常見組合：TTL + Event-based — event 主動刪 + TTL 當 safety net。

TTL trade-off：短 TTL (30s) → 資料新，但 hit ratio 低 → DB 壓力大；長 TTL (24hr) → hit ratio 高，但資料 stale 久。

StampedeCache Stampede 是什麼？三種解法？
Hot key TTL 到期 → 1000 個 request 同時 cache miss → 全部打 DB → DB 爆了
  1. Lock：第一個 miss 拿 lock 查 DB，其他人等
  2. 隨機 TTL：TTL = 300s + random(0-60s) 分散過期
  3. Stale-while-revalidate：先回舊資料，背景更新
🔗跨主題連結：Stampede 的 single-flight lock = Day 27 熱 key 的 service 本地 cache、thundering herd / singleflight 同一個家族 — 把「同時全部回源」收斂成「一個回源、其他等」。看到「一堆人同時撲同一個資源」就想這招。
▸08 CDN Deep Dive

邊緣快取 Pull / Push / Origin Shield

回想Pull CDN vs Push CDN：運作、第一次請求、適合、維護？
Pull CDNPush CDN
運作第一次請求時 edge 去 origin 拿主動上傳到所有 edge
第一次請求慢（回 origin）快（已在 edge）
適合流量大、內容多內容少、可預測（CSS/JS）
維護簡單，CDN 自動管複雜，要管上傳邏輯
回想Origin Shield 是什麼？解決什麼問題？

Edge 和 Origin 中間的共用 cache 層，避免多個 Edge 同時打 Origin（類似 cache stampede 概念）。

User → Edge (miss) → Origin Shield (shared cache) → Origin Server
🔗跨主題連結：Pull CDN（edge 自己去 origin 拿、本地 cache、配 TTL）= Day 14 JWKS pull 模型、DNS resolver cache 同型。三件套：pull + 本地快取 + TTL，代價都是「新鮮度 ≈ TTL」。
深掘Origin Shield 是 edge 和 origin 中間的共用 cache。這跟 cache stampede 的 lock 解法是不是同一招？差在哪？蘇格拉底深掘 · 非原始筆記

同精神、不同層。兩者都是「收斂回源」：別讓 N 個下游同時打同一個上游。差別在誰收斂、收斂什麼。

Stampede lock 是在單一 cache 節點內收斂同一個 key 的並發 miss（只放一個 request 查 DB）。Origin Shield 是在跨地理的多個 edge 之間多加一層共用 cache：原本 50 個 edge 各自 miss 就打 50 次 origin，現在它們先打同一個 shield，shield miss 才打 origin 一次。一個收斂「同節點同 key 的並發」，一個收斂「多節點對同內容的回源」。本質都是「在撲向源頭的路上插一個收斂點」。

▸09 CDN Invalidation

三種刷新 為什麼 Versioned 最香

回想CDN 三種 invalidation：做法 / 速度 / 費用？
Strategy做法速度費用
TTL-based設 Cache-Control: max-age，等過期慢（小時）免費
Purge (Invalidation API)呼叫 API 刪除所有 edge 上的特定 path快（5-15 分鐘）$$（CloudFront: 前 1000 次/月免費，之後按 path 收費）
Versioned URLs檔名帶 hash（style.a1b2c3.css）即時免費
Best PracticeStatic assets 和 HTML pages 各用什麼組合？為什麼 Versioned 優於 Purge？
  • ▸Static assets (CSS/JS/images)：Versioned URLs + 長 TTL（1 年） — build tool 產生 hash filename → 新版本 = 新 URL → 不用 invalidate
  • ▸HTML pages：短 TTL (60s) + Purge 當 fallback — HTML 裡引用新 asset URL，所以 user 自動拿到新檔案

為什麼 Versioned URLs 優於 Purge：

  1. 即時生效（Purge 要 5-15 分鐘 propagation）
  2. 免費（Purge 要付錢）
  3. 保證不會拿到舊版（新 filename = 不可能 cache hit 到舊的）
# CloudFront Invalidation API
aws cloudfront create-invalidation \
  --distribution-id E1234 \
  --paths "/*"    # wildcard 算 1 path，省錢
🔗跨主題連結：Versioned URL（content-hash 檔名）= Day 27 的 immutable mapping：內容不變 → URL 不變 → cache 永久有效、零 invalidation。同一個「不可變 = 免失效」原則，一個用在短碼、一個用在資產檔名。
深掘Versioned URL 即時又免費，那為什麼 HTML 本身不也用 versioned 檔名，還要短 TTL + Purge？蘇格拉底深掘 · 非原始筆記

因為 HTML 是入口，它的 URL 必須穩定 — 使用者打 example.com/、書籤、SEO 連的都是固定網址，你不能每次部署就換首頁網址。

所以策略是兩段式：固定 URL 的 HTML 吃短 TTL（讓它很快更新），HTML 內容裡引用的 CSS/JS/圖用 versioned URL。部署後 HTML 一更新（短 TTL 過後），裡面就指向新的 app.3f9a.js，使用者自動連到新 asset。會變的入口吃短 TTL、不可變的資產吃 versioned，各取所長。往回扣 best practice 組合。

▸12 🔴 My Mistakes & Misconceptions

糾錯 先看錯的，再翻正解

✗ 當初的錯先刪 cache 再寫 DB 沒問題
想一下正確理解 → 點開
✓ 正確理解先刪 cache 再寫 DB 有 race condition：刪完 cache 後、寫 DB 前，另一個 thread 可能讀到舊的 DB 值寫回 cache。我沒考慮到 concurrent read 在 delete 和 write 之間的時間窗口。
✗ 當初的錯Write-Behind 的風險是 DB shutdown
想一下正確理解 → 點開
✓ 正確理解風險是 Cache shutdown — 資料先寫在 cache 還沒 flush 到 DB，cache 掛了資料就不見了。我搞反了哪個元件掛掉才是危險的。
✗ 當初的錯不懂 ctx 為什麼要傳給 Redis
想一下正確理解 → 點開
✓ 正確理解ctx 是操作的「控制遙控器」— Redis 是網路呼叫，可能會卡住，ctx 讓你設 timeout 或取消。PoC 用 context.Background()（不設限），生產用 context.WithTimeout。我沒意識到 Redis call 是網路操作，需要超時控制機制。
✗ 當初的錯Cache-aside 的 nested if/else 寫錯（少了內層 if err == nil）
想一下正確理解 → 點開
✓ 正確理解兩層 if：外層判斷 cache 開/關，內層判斷 hit/miss。漏掉內層 = hit 和 miss 的 code 沒有分開。我不熟悉 Go 的 nested if/else 結構，把兩個判斷混在一起。
